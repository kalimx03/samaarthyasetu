## Technical Questions

### Q1: Can an LLM actually solve any task without rule-based help?

**Yes.** Run this to watch it live:

```bash
export HF_TOKEN="your-key"
export API_BASE_URL="https://api.openai.com/v1"
export MODEL_NAME="gpt-4o-mini"
python inference.py --mode llm --task task_001 --demo
```

In `--mode llm`, the agent starts from `env.reset()` and every single action is generated by the LLM from the conversation history. The rule scripts are not consulted at all. The only fallback is a `finalize_task` safety net if the LLM returns malformed JSON on both of 2 retries (which does not happen with gpt-4o-mini or gpt-4o on this task).

The output shows a `[llm]` or `[llm-safety-fallback]` label on every step so you can verify this claim in real time.

### Q2: What is the difference between the three inference modes?

| Mode | LLM called? | Rules used? | When to use |
|------|-------------|-------------|-------------|
| `rules` | Never | Always | Reproducible evaluation, CI, no API key |
| `llm` | Every step | Never (safety fallback only on parse error) | Demo, live judging, showing agent reasoning |
| `hybrid` | Every step (preferred) | On LLM parse failure | Production robustness |

### Q3: How does the matching engine work? What prevents it from being trivial?

The composite score is:

```
score = 0.60 × skill_jaccard(candidate, job)
      + 0.25 × accommodation_coverage(candidate, job)
      + 0.15 × language_overlap(candidate, job)
```

`skill_jaccard` uses set intersection/union, not substring matching, so `"Coding"` and `"code"` are distinct. `accommodation_coverage` checks whether each of the candidate's specific disability needs (e.g. `"Wheelchair access"`) is covered by the employer's listed accommodations — this is not trivial because it's a semantic keyword overlap, not an exact string match.

For task_001, cand_003 (Arjun Nair, React/JS developer with locomotor disability) scores **0.95** against job_002 (Infosys BPM Junior Frontend Developer). The next-best match is 0.23 — a 4× gap. This makes the task meaningful without being ambiguous.

### Q4: Is the reward function sparse or dense?

**Dense.** Partial-progress signals fire on the first step each sub-goal is achieved:

| Signal | Bonus | Fires when |
|--------|-------|------------|
| `candidate_loaded` | +0.04 | First `get_candidate` succeeds |
| `match_computed` | +0.08 | `match_candidate_to_job` executes |
| `high_quality_match` | +0.04 | Match score ≥ 0.75 |
| `interview_confirmed` | +0.09 | Interview scheduled with correct employer |
| … 8 more | … | … |

Each signal fires **at most once per episode**. The step efficiency component uses a smooth cosine decay (`0.5 × (1 + cos(π × steps/max_steps))`) so there is gradient signal throughout the episode, not just at the end. This is documented in `docs/REWARD_DESIGN.md` and tested in `tests/test_reward_shaping.py` (501 lines of tests).

### Q5: Why is this an OpenEnv environment and not just a FastAPI wrapper?

OpenEnv defines three interface methods: `reset()`, `step()`, `state()`. Our environment:

- `reset(task_id)` returns a fully typed `Observation` (Pydantic v2) with task description, required actions, and reward design
- `step(action)` returns `(Observation, reward, done, info)` — the canonical RL interface
- `state()` returns a typed `State` snapshot

The FastAPI server in `server/app.py` exposes these three methods as REST endpoints (`POST /reset`, `POST /step`, `GET /state/{task_id}`) so any HTTP client — including Meta's evaluation harness — can drive the environment.

### Q6: What does "deterministic graders" mean and why does it matter?

The graders in `samaarthya_ops_env/graders/task_graders.py` are pure functions:

```
score, details = grade_task_001(state_history, final_state)
```

They contain no randomness, no external calls, no time-dependent logic. The same final state always produces the same score. This is required by OpenEnv's automated evaluation pipeline — if graders were probabilistic, scores would vary between runs and make leaderboard ranking unreliable.

### Q7: How scalable is this environment?

Each environment instance is a single Python object — no I/O, no network. It runs a full task in under 1 second on commodity hardware. Multiple instances can be parallelised via `multiprocessing` or `Ray`. The Docker container runs on 2 vCPU / 2 GB RAM (well within HuggingFace Spaces free tier).

### Q11: What is the placement conflict task (task_003) based on?

This is based on a real and widespread problem in India's disability employment sector: NGOs are incentivised to report high placement numbers and sometimes mark candidates as "placed" before the employer has confirmed a hire. The candidate, meanwhile, is still searching. When the government follows up, records disagree.

Our environment simulates this 3-way conflict (NGO record vs. employer record vs. follow-up record) and requires the agent to classify the conflict type (`ngo_employer_mismatch`), choose a resolution (`schedule_fresh_interview`), and update the public ward dashboard — creating a verifiable paper trail.
